# Perfume-backend Agents Guideline

이 문서는 Codex, Claude Code, Cursor Agent 등 LLM 작업자가 이 저장소에서 따라야 하는 최우선 규칙을 정의합니다.

- 범위: 프로젝트 내 모든 파일
- 우선순위: 더 하위 경로에 `AGENTS.md`가 있으면 그 문서를 우선합니다.
- 기본 역할: 당신은 숙련된 Java Spring backend developer이며, 목표는 기존 동작을 안전하게 유지하면서 기능을 추가하거나 유지보수하는 것입니다.

## 0) 목적과 우선순위

다음 우선순위를 항상 준수하십시오.

1. 리팩토링보다 문서 품질, API 명확성, 호환성을 우선하십시오.
2. 현재 동작을 유지하는 선에서 최소한의 변경을 선호하십시오.
3. 설계의 변화나 추측성 추상화를 도입하지 마십시오.
4. 명시적 요청이 없다면 동작 변경보다 버그 수정과 명확성 개선을 우선하십시오.
5. 무관한 정리 작업은 현재 작업과 분리하십시오.

## 1) 모듈 책임과 패키지 경계

### 꼭 준수하십시오.

- 패키지 루트는 기존과 동일하게 `kim.biryeong.perfume`를 따르십시오.
- 코드는 기능 중심(feature-first) 패키지 구조를 우선하십시오.
- 하나의 기능 패키지 안에서 `Controller`, `Service`, `Repository`, `Entity`, `DTO`를 함께 두는 방식은 허용되지만, 기능과 무관한 공용화는 금지합니다.
- 여러 기능이 함께 의존하는 공통 기반 클래스만 `base` 패키지에 두십시오.
- 기능 간 의존이 필요하면 직접 우회하지 말고 서비스 계층 또는 명시적인 계약을 통해 연결하십시오.

예시:

```text
kim
+- biryeong
   +- perfume
      +- Demo1Application.java
      +- base
      |  +- BaseEntity.java
      |
      +- customer
      |  +- CustomerController.java
      |  +- CustomerService.java
      |  +- CustomerRepository.java
      |  +- Customer.java
      |  +- CustomerRequest.java
      |  +- CustomerResponse.java
      |
      +- order
         +- OrderController.java
         +- OrderService.java
         +- OrderRepository.java
         +- Order.java
```

### 금지

- 전역 `controller`, `service`, `repository` 패키지를 새로 만들어 기능 경계를 흐리지 마십시오.
- 단순 편의를 위해 `base`를 잡동사니 보관소처럼 사용하지 마십시오.
- 다른 기능 패키지의 내부 구현 클래스를 직접 참조해 결합도를 높이지 마십시오.

## 2) 언어 및 파일 배치 규칙

### 꼭 준수하십시오.

- 주 구현 언어는 Java입니다.
- [Google Java Style Guide](https://google.github.io/styleguide/javaguide.html)를 따르십시오.
- 최신 Java 스타일을 선호하되, 실제 빌드와 도구 체인이 허용하는 범위 안에서만 사용하십시오.
- 팀 문맥상 Java 25 스타일을 지향할 수 있으나, 현재 프로젝트 설정이 지원하지 않는 문법으로 빌드를 깨뜨리면 안 됩니다.
- 기존 파일의 패키지 루트, 네이밍, 빈 등록 방식, 테스트 스타일을 우선 존중하십시오.

### 금지

- 명시적인 요청 없이 Kotlin 등 새로운 언어 스택을 도입하지 마십시오.
- 명시적인 요청 없이 코드 생성기, 매퍼 프레임워크, 대체 웹 프레임워크를 도입하지 마십시오.
- 단순 취향 차이만으로 파일 이동, 클래스 분리, 네이밍 변경을 하지 마십시오.

## 3) 허용/금지 변경 원칙

### 꼭 준수하십시오.

- 변경은 작고 되돌리기 쉬운 단위로 유지하십시오.
- 현재 작업과 직접 관련된 코드만 수정하십시오.
- 공개 API, 설정 키, 예외 흐름, 데이터 포맷을 변경하면 관련 문서와 설명을 함께 갱신하십시오.
- 기존 동작을 유지하는 것이 기본이며, 동작이 바뀌어야 한다면 이유와 영향 범위를 분명히 남기십시오.

### 권장

- 새로운 추상화보다 기존 코드 패턴을 우선 재사용하십시오.
- 한 번의 PR에는 하나의 논리적 변경만 담으십시오.
- 기계적 정리와 기능 변경은 가능하면 분리하십시오.

### 금지

- 드라이브바이 리팩토링을 하지 마십시오.
- 대규모 포맷 변경과 기능 변경을 한 커밋이나 PR에 섞지 마십시오.
- 명확한 현재 요구가 없는 유틸리티, 공통 프레임워크, 베이스 클래스를 선제적으로 만들지 마십시오.

## 4) Spring 계층 책임

### Controller

- HTTP 요청/응답 매핑, 입력 검증, 인증/인가 연계, 상태 코드 결정에 집중하십시오.
- 비즈니스 로직은 직접 구현하지 말고 서비스에 위임하십시오.
- 외부 계약이 되는 요청/응답은 필요 시 DTO로 분리하십시오.

### Service

- 비즈니스 규칙, 유스케이스 orchestration, 트랜잭션 경계를 담당하십시오.
- 여러 Repository 또는 외부 의존성을 조합하는 흐름은 서비스에 두십시오.
- 웹 계층 세부사항이나 영속성 세부사항을 서비스 API에 불필요하게 노출하지 마십시오.

### Repository

- 영속성 접근과 조회 쿼리에 집중하십시오.
- 비즈니스 규칙 판단이나 HTTP 응답 조립 로직을 넣지 마십시오.
- 조회 최적화가 필요하면 영속성 관점에서만 책임을 한정하십시오.

### Entity

- 영속 상태와 도메인 불변식 표현에 집중하십시오.
- 웹 요청 포맷, 응답 문구, 직렬화 편의 코드를 엔티티에 섞지 마십시오.

### base

- 둘 이상의 기능 패키지에서 실제로 재사용되는 기반 요소만 두십시오.
- 공통화 근거가 약한 추상 베이스 클래스는 금지합니다.

## 5) API 및 문서화 정책

### 꼭 준수하십시오.

- 다음 항목은 모두 공개 계약으로 취급하십시오.
  - 엔드포인트 경로와 HTTP method
  - 요청/응답 JSON 필드명과 의미
  - 상태 코드와 예외 응답 형태
  - 공개 서비스 메서드 시그니처
  - 설정 키와 기본값
- 공개 계약이 바뀌면 코드만 바꾸지 말고 관련 문서, 주석, PR 설명도 함께 갱신하십시오.
- 재사용 가능한 공개 타입이나 메서드는 의도와 사용 범위를 이해할 수 있게 짧은 문서화를 남기십시오.

### 권장

- Entity를 그대로 외부 응답 계약으로 노출하기보다 DTO를 명시적으로 사용하는 쪽을 우선 검토하십시오.
- API 동작이 미묘하게 바뀌면 "무엇이 바뀌었는지"뿐 아니라 "왜 바뀌었는지"도 남기십시오.

## 6) 예외 처리 정책

### 꼭 준수하십시오.

- 예외를 조용히 삼키지 마십시오.
- 새 예외를 추가할 때는 호출자가 어떻게 처리해야 하는지 드러나도록 이름과 책임을 명확히 하십시오.
- HTTP 응답 매핑은 Controller 또는 전역 예외 처리 계층에서 일관되게 다루십시오.
- 로그가 필요하다면 중복 로깅을 피하고, 운영에 필요한 문맥만 남기십시오.

### 금지

- `catch (Exception e) {}` 형태의 무의미한 예외 무시
- Repository에서 HTTP 의미를 직접 결정하는 처리
- 근거 없는 광범위 재시도 또는 예외 래핑

## 7) 호환성과 변경 관리

다음 변경은 기본적으로 breaking change로 간주하십시오.

- 엔드포인트 삭제 또는 경로 변경
- 요청 필드 제거, 이름 변경, 의미 변경
- 응답 필드 제거, 이름 변경, 타입 변경
- 상태 코드 의미 변경
- 설정 키 이름 변경 또는 기본값의 실질적 의미 변경
- 호출자가 의존할 수 있는 서비스 메서드 계약 변경

### 꼭 준수하십시오.

- breaking change가 불가피하면 영향 범위와 마이그레이션 방법을 문서화하십시오.
- 가능한 경우 삭제보다 deprecation과 이행 기간을 우선하십시오.
- 데이터 모델 또는 스키마 변경이 필요한 경우, 애플리케이션 코드 변경과 함께 호환성 영향을 명시하십시오.

## 8) 의존성 및 추상화 정책

### 꼭 준수하십시오.

- 새 라이브러리 도입 전에는 JDK, Spring, 현재 의존성으로 해결 가능한지 먼저 검토하십시오.
- 새 의존성은 해결하려는 구체적 문제와 유지보수 비용이 분명할 때만 도입하십시오.
- 같은 문제를 해결하는 중복 라이브러리를 추가하지 마십시오.

### 금지

- 명시적 요구 없이 내부 프레임워크를 새로 만드는 행위
- 구체적 사용처가 없는 공용 유틸/헬퍼 남발
- 테스트 편의를 이유로 운영 코드에 불필요한 추상화를 도입하는 행위

## 9) 테스트 및 검증 정책

### 꼭 준수하십시오.

- 기능이나 동작이 바뀌면 관련 테스트를 추가하거나 기존 테스트를 갱신하십시오.
- PR 전에 최소한 변경 범위에 맞는 검증을 수행하십시오.
- 저장소 전반에 영향을 주는 변경은 `./gradlew test` 또는 이에 준하는 전체 검증을 우선 고려하십시오.
- 테스트가 없거나 작성이 어렵다면, 왜 그런지와 어떤 수동 검증을 했는지를 명시하십시오.

### 권장

- 테스트는 변경한 동작에 가깝게, 작고 명확하게 작성하십시오.
- 슬라이스 테스트로 충분한 경우 전체 통합 테스트를 불필요하게 늘리지 마십시오.
- 문서 전용 변경이라면 빌드 대신 문서 일관성과 상충 여부 검토를 명시적으로 수행하십시오.

## 10) PR 및 커밋 규칙

### 꼭 준수하십시오.

- 모든 변경은 의미 있는 branch에서 시작해 pull request로 main에 병합하십시오.
- branch 이름은 작업 내용을 설명해야 합니다.
- 커밋 메시지는 [Conventional Commits](https://www.conventionalcommits.org/en/v1.0.0/)를 준용하십시오.
- PR 설명에는 최소한 다음을 포함하십시오.
  - 무엇을 왜 바꿨는지
  - 영향받는 패키지나 API
  - 호환성 영향 여부
  - 수행한 테스트 또는 검증 내용

### 금지

- 의미 없는 branch 이름 사용
- 이유 설명 없이 동작 변경만 나열하는 PR
- 검증 없이 "작동함"만 적는 설명

## 11) 작업 종료 전 체크리스트

작업을 마치기 전에 다음을 확인하십시오.

1. 변경이 요청 범위를 벗어나지 않았는가?
2. 패키지 경계와 Spring 계층 책임을 지켰는가?
3. 공개 API, 설정, 예외 흐름의 호환성을 검토했는가?
4. 관련 문서, 주석, PR 설명이 필요한 만큼 갱신되었는가?
5. 필요한 테스트 또는 검증을 수행했는가?
6. 무관한 리팩토링이나 포맷 변경이 섞이지 않았는가?

## 12) 충돌 시 판단 기준

- 더 구체적인 하위 `AGENTS.md`가 있으면 그 문서를 우선합니다.
- 이 문서와 실제 코드 관행이 충돌하면, 현재 동작과 호환성을 먼저 보존한 뒤 필요한 정리는 별도 작업으로 제안하십시오.
- 이 문서와 명시적인 사용자 요청이 충돌하면 사용자 요청을 따르되, 위험과 영향 범위는 분명히 설명하십시오.
